Q1: 为什么选择 Qt6 + Python 的混合架构，而不是纯 Python 或纯 C++？
--------------------------------------------------------------------------------

  答：这是一个"各取所长"的设计决策。

  选择 Qt6 做前端的原因：
  - Qt6 的跨平台 GUI 能力成熟，QProcess 子进程管理方便
  - C++ 编译为原生 exe，启动速度快，用户体验好
  - Qt6 的 signal/slot 机制让异步操作（如转换进度回调）实现简洁

  选择 Python 做后端的原因：
  - Python 生态中文档处理库非常成熟——pdf2docx 封装了复杂的 PDF 解析逻辑，
    PyMuPDF 提供了完整的 PDF 底层 API，python-docx 可以细粒度操作 Word 文档
  - 如果用纯 C++ 实现同样的功能，需要集成 poppler/libreoffice 等重型 C 库，
    开发周期和复杂度会成倍增长
  - Python 脚本可以独立调试和测试，不依赖 GUI 启动，开发效率高

  通信方式的选择：
  - 通过 QProcess 调用 Python 子进程，JSON 通过 stdout 传递结果
  - 这种方式比嵌入 Python 解释器（Python/C API）更简单，不引入 CPython ABI 兼容问题
  - 比 HTTP 本地服务更轻量，不需要管理端口和网络

  代价：每次转换需要启动 Python 子进程，有约 100-300ms 的冷启动开销。
  对于文档转换这种低频操作来说完全可以接受。


Q2: 你的 PdfWriter 自研排版引擎是怎么实现的？为什么不用现成的库？
--------------------------------------------------------------------------------

  答：PdfWriter 是我为解决"Word/Markdown/JSON → PDF"这条路径写的轻量
  排版引擎，核心原因是：

  为什么要自研：
  - 现有库的局限性：用 docx2pdf 做 Word→PDF 依赖本机安装的 Word，如果用户的
    Word 版本不兼容或没装 Word 就会失败。fitz 本身没有"把文字排成段落的
    高级 API"，只提供 TextWriter 底层的文本写入能力。
  - 我需要一个不依赖外部程序的纯 Python 方案作为 fallback。

  实现原理（80行代码）：
  1. 字体加载：_get_cjk_font() 按优先级自动检测系统中的 CJK 字体文件
     （微软雅黑→宋体→黑体），确保中文正常渲染

  2. 文本宽度精确度量：调用 font.text_length(text, fontsize=fs) 获取文本
     在指定字号下的实际渲染宽度（像素），这是自动换行的基础

  3. 自动换行算法：当 text_length > 可用宽度时，从末尾逐字符缩短文本，
     找到最后一个可放入当前行的切分点，然后将剩余文本递归写入下一行

  4. 自动分页：维护 self.y 坐标，当累计高度超过 780pt（A4 页面可用高度）
     时，调用 TextWriter.write_text(page) 将缓冲区刷入当前页，然后创建
     新页并重置坐标

  5. 样式控制：write() 接收 fs（字号）、indent（缩进）、dy（行间距）三个
     参数，实现标题/段落/引用/列表的差异化排版

  不足之处：当前只支持纯文本排版，不支持图片嵌入、表格渲染、页眉页脚等
  高级功能。这些在"Word→PDF"路径中由 docx2pdf 优先处理，fitz fallback
  只在 docx2pdf 不可用时才走。


Q3: TCP 粘包问题你了解吗？这个项目中有没有类似的"数据边界"处理？
--------------------------------------------------------------------------------

  答：这个问题问得很好。虽然我这个项目是进程间通信（完整 JSON 一次性输出），
  没有 TCP 粘包问题，但我在另一个 Qt 网络聊天项目中深入处理过 TCP 粘包。

  核心思路：TCP 是流式协议，发送方 write() 的次数和接收方 read() 的次数
  没有对应关系。解决方案是在应用层定义消息边界。

  常见的三种处理方式：
  1. 固定长度报文（不灵活，浪费带宽）
  2. 分隔符方式（如 HTTP 用 \r\n\r\n，但如果数据本身包含分隔符需要转义）
  3. 长度前缀方式（我采用的方案）：每条消息头部固定 4 字节存放消息体长度，
     接收方先读 4 字节获知长度，再读对应长度的数据，不够就继续等待

  如果把这个思路对应到当前项目：我的 JSON 响应是一次性 print 的完整 JSON
  对象，天然具有边界（{} 闭包）。但如果未来改为流式处理大文件（边转换
  边传输），就需要设计类似的分帧协议。


Q4: PDF 转 Word 的过程中，你是怎么处理复杂排版（表格、图片、多栏）的？
--------------------------------------------------------------------------------

  答：坦诚地说，pdf2docx 库本身对表格和简单排版的支持还不错，但对多栏布局、
  扫描件 PDF（图片型 PDF）的转换效果确实有限。我的处理策略分几层：

  1. 正常 PDF（从 Word 生成的）：pdf2docx 能较好地还原文字、表格和基础格式
  2. 复杂排版 PDF：pdf2docx 尽力保留文字内容和位置信息，布局可能有偏移
  3. 扫描件/图片型 PDF：当前无法处理——这是我规划中集成 OCR（pytesseract）
     的扩展方向

  降级方案：
  - PDF → Markdown 路径：我做的是纯文本提取+格式推断（按字号判标题、
    按 bold 判加粗），牺牲布局完整性换取可编辑性
  - PDF → JSON 路径：我保留了完整的结构化信息（四级层级+颜色+坐标），
    这种结构化数据可以用于下游的文档分析或自定义渲染，本质上是做了一个
    "PDF 反序列化"

  客观上，PDF 是"打印格式"而非"编辑格式"，完美还原本身就非常困难——
  即使 Adobe Acrobat 的付费版本也不完美。所以合理降低用户预期、提供
  多种输出格式让用户选择，是这个项目务实的设计思路。


Q5: 如果要你把这个项目做成一个 Web 服务，你会怎么设计？
--------------------------------------------------------------------------------

  答：我会采用 FastAPI + Celery 异步任务队列的架构：

  架构设计：
  ┌─────────────┐     HTTP/JSON     ┌──────────────┐     Redis     ┌──────────┐
  │  Web 前端    │  ───────────────→ │  FastAPI 服务  │  ──────────→ │  Celery   │
  │  (Vue/React) │  ←─────────────── │  (同步接口)    │  ←────────── │  Worker   │
  └─────────────┘                   └──────────────┘              └──────────┘
                                          │                            │
                                          │ 文件存储                     │ 转换引擎
                                          ▼                            ▼
                                    ┌──────────┐              ┌──────────────┐
                                    │  OSS/MinIO│              │  convert.py  │
                                    └──────────┘              └──────────────┘

  关键设计点：
  1. 文件上传：前端上传文件 → FastAPI 接收 → 存到 OSS/MinIO → 返回 file_id
  2. 异步转换：转换请求先入库（status=pending），推入 Celery 队列，
     立即返回 task_id。用户轮询或 WebSocket 获取进度
  3. 文件下载：转换完成后生成临时下载链接（有过期时间），用户下载后清理
  4. 限流和并发控制：Celery 设置 concurrency 限制并行转换数，
     避免 PDF 大文件撑爆服务器内存
  5. 当前的 convert.py 几乎可以直接复用为 Celery task 函数，
     只需修改输入路径（从本地路径改为 OSS 下载后的临时路径）

  核心改造量：把 sys.argv 入口改成函数调用，文件 I/O 层的路径适配一下即可。
  这是我当初把转换逻辑全部独立在 convert.py 中的好处——换一个调用方
  （Qt6 → FastAPI）改动很小。


Q6: 你在项目中遇到的最难的技术问题是什么？怎么解决的？
--------------------------------------------------------------------------------

  答：最难的问题是 CJK 中文字体在 PyMuPDF 中的渲染问题。

  问题描述：
  用 fitz.TextWriter 写入中文文本时，如果不指定字体，默认的 Helvetica 字体
  会把中文渲染成方框（tofu）。PyMuPDF 的 TextWriter 需要显式传入 fitz.Font
  对象。

  解决过程：
  1. 首先调研了 PyMuPDF 的字体机制：它支持使用系统字体文件（.ttc/.ttf）
     创建 fitz.Font 对象

  2. 写了 _get_cjk_font() 函数，按优先级自动检测 Windows 系统中的 CJK 字体：
     微软雅黑(msyh.ttc) → 宋体(simsun.ttc) → 黑体(simhei.ttf) → Helvetica 回退
     这样在绝大多数中文 Windows 系统上都能自动找到可用字体

  3. 更进一步，用 font.text_length() 精确计算中文字符的渲染宽度。
     中文字符是等宽的二倍于英文，但不同字体的实际渲染宽度不同，
     用 text_length 才能精确计算，避免自动换行时截断位置不准确

  这个问题的解决让我深入理解了：
  - 字体渲染的 cross-platform 兼容性问题
  - "检测→回退"的优雅降级策略
  - 为什么商业 PDF 库（如 iText）需要内嵌字体


Q7: 你的 PDF 转 JSON 功能提取了哪些信息？有什么应用场景？
--------------------------------------------------------------------------------

  答：我提取了 PDF 的四级层级结构：

  pages[] → blocks[] → lines[] → spans[]
     │          │          │         └── text(文本), font(字体), size(字号),
     │          │          │             color(#RRGGBB 颜色), bbox([x,y,w,h] 坐标)
     │          │          └── spans[] 列表
     │          └── type: "text" | "image", bbox 坐标
     └── page_number, width, height

  除此之外还有文档级元数据：total_pages, file_size

  应用场景：
  1. 文档内容分析：基于坐标的段落合并（相邻 spans 判断是否同一段落）、
     标题层级识别（按字号排序）
  2. 全文检索系统：把 PDF 内容提取为结构化 JSON，导入 Elasticsearch
     建立索引，支持按页定位
  3. 数据标注：PDF 版面分析的数据预处理（标注每个 span 的位置和样式）
  4. 格式转换中间层：其他格式的转换都可以先从 JSON 这个"通用中间格式"
     出发，只需要写一次"JSON → 目标格式"的转换逻辑

  这是一个"结构化存储"思想的体现——把非结构化的 PDF 变成结构化的 JSON，
  后续的所有处理都基于结构化数据，而不是每次都去解析 PDF。

  2. convert.py 单文件 733 行：虽然功能完整，但所有转换函数堆在一个文件里。
     更好的做法是按转换方向拆分为独立的 handler 模块，便于测试和维护

  3. 缺少单元测试：目前全靠手动命令行测试。应该引入 pytest，对每条转换
     路径写标准输入/输出的测试用例

  4. 错误处理可以更细粒度：当前是统一的 try/except 捕获所有异常，导致
     用户看到的错误信息可能是 Python 的底层报错。应该区分"文件不存在"
     "文件损坏""不支持该 PDF 版本"等情况，给出更精确的提示

  5. 大文件内存问题：PDF 的合并和拆分都是全量加载到内存，对于几百 MB 的

  6. Markdown 表格支持不完整：当前只做分隔行过滤，没有真正做表格渲染。
     应该引入 tabulate 或手写表格解析器

  主动指出不足反而能展示你的代码质量意识和工程成熟度。


Q9: 你对 PyMuPDF、pdf2docx、python-docx 这几个库的底层原理了解多少？
--------------------------------------------------------------------------------

  答：

  PyMuPDF (fitz)：
  - 底层基于 MuPDF（Artifex 公司的开源 PDF 引擎，C 语言编写）
  - fitz 通过 Python C 扩展封装 MuPDF 的 API
  - PDF 页面渲染：MuPDF 将 PDF 的绘图指令解析为 display list，再光栅化
  - 文本提取：通过分析 PDF 的内容流(Content Stream)中的文本操作符
    （Tj, TJ, Tm 等），提取字符和位置信息
  - 这就是为什么 PDF 中的文字不是按"段落"而是按"span"（同一字体/
    大小的连续字符片段）组织的——因为 PDF 本身就没有段落概念

  pdf2docx：
  - 原理：解析 PDF 页面中的每个元素（文本块、图片、线条），分析它们的
    坐标位置和样式属性，然后按照 Word 文档的布局规则重建为 python-docx
    的段落和表格对象
  - 关键挑战：PDF 没有"段落""表格"的显式概念，需要通过坐标聚类来判断
    哪些文本属于同一段落、哪些线条构成表格边框

  python-docx：
  - 底层操作 .docx 文件（本质是 ZIP 包内的 XML 文件集合）
  - 主要操作 word/document.xml（文档正文）、word/styles.xml（样式定义）
  - 通过 lxml 库解析和修改 XML，遵循 OOXML 标准


Q10: 如果让你重新设计这个项目，你会做什么改变？
--------------------------------------------------------------------------------

  答：

  1. 分层架构：当前 convert.py 是"脚本式"的，所有逻辑混在一起。
     重新设计会分成清晰的 layers：
     - CLI Layer（命令行参数解析，用 argparse 替代手写 sys.argv 路由）
     - Service Layer（转换服务，包含统一的错误处理和日志）
     - Converter Layer（每种转换方向一个独立的 converter 类）
     - Renderer Layer（PdfWriter 等渲染器）

  2. 引入中间表示层（IR）：定义一种统一的"文档中间格式"（类似 AST），
     所有格式先解析为 IR，再从 IR 渲染为目标格式。这样 N×N 的转换矩阵
     变成 2N 的复杂度（N 种格式 × 2 个方向）

  3. 用 argparse 替代手写参数解析：当前 split 命令的参数位置依赖容易出错，
     argparse 的子命令模式（subparsers）更健壮

  4. 增加进度回调：当前大文件转换无进度反馈，可以通过 stderr 输出进度
     JSON，QProcess 同时监听 stdout 和 stderr

  5. 考虑容器化部署：用 Docker 打包 Python 环境，解决"用户没装 Python"
     或"依赖版本冲突"的问题
